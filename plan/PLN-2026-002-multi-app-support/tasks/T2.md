## Context

The permissions enforcer currently takes global and project configs to build the deny list. Per D3, a third layer (app-level deny) must be added. The enforcer's `NewEnforcer` function gains a third parameter for the app-specific deny list. The merge order remains: global -> project -> app, each layer can only restrict further.

## Instructions

1. **Update `NewEnforcer` in `internal/permissions/enforcer.go`:**
   - Change signature from `NewEnforcer(global config.GlobalConfig, project config.ProjectConfig)` to `NewEnforcer(global config.GlobalConfig, project config.ProjectConfig, appDeny []string)`
   - After applying project denials, apply app denials with the same pattern:
     ```go
     for _, cmd := range appDeny {
         if allowed[cmd] {
             deniedBy[cmd] = "app"
         }
         allowed[cmd] = false
     }
     ```
   - The `deniedBy` map now has three possible values: "global", "project", "app"

2. **No changes needed to `types.go`** — the `PermissionDeniedError.DeniedBy` field is already a string and can hold "app".

3. **Verify compilation:** Run `go build ./internal/permissions/...` mentally. The callers of `NewEnforcer` (in `agent.go` and `doctor.go`) will be updated in S2. For now, the callers will fail to compile — this is expected and will be fixed in T6/T8.

   **Important:** To keep S1 compilable, add a temporary compatibility wrapper or update the two call sites to pass an empty `[]string{}` as the third argument. The call sites are:
   - `internal/cli/agent.go` line 74: `permissions.NewEnforcer(*globalCfg, *projectCfg)`
   - `internal/cli/doctor.go` line 224: `permissions.NewEnforcer(*globalCfg, *projectCfg)`

   Update both to: `permissions.NewEnforcer(*globalCfg, *projectCfg, []string{})` — this preserves existing behavior.
